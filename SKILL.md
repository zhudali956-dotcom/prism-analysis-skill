# 棱镜分析法深度研究

> 棱镜分析由用户与 Claude 共同打磨完成，融合了多维搜索体系与叙事分析框架。

---

## 核心原则

棱镜分析的核心是三个关键词：**拆开看、追到底、讲出来**。

- **拆开看** — 不满足于单一角度的描述，从多维度审视研究对象
- **追到底** — 不满足于表层信息，逐层追问直到源头
- **讲出来** — 不满足于罗列事实，用叙事把判断交给读者

最终产出一份**排版精美的 PDF 研究报告**和对应的 **Markdown 源文件**。

---

## 工作流程概览

```
第一层：搜索层 — 信息勘探与提纯
    │ 多轮搜索 + 可信度标签 + 交叉验证
    ▼
第二层：分析层 — 四维解构
    │ 起源 → 分野 → 格局 → 推演
    ▼
第三层：输出层 — 叙事呈现
    │ 故事化写作 + HTML/MD/PDF 三输出
```

## 第一层：搜索层 — 信息勘探与提纯

### 环境准备

1. **PDF 转换脚本**：本 Skill 自带 `scripts/md_to_pdf.py`（基于 WeasyPrint），用于将 Markdown 转为排版精美的 PDF。确保依赖已安装：`pip install weasyprint markdown --break-system-packages`。如果环境不支持 WeasyPrint，则只输出 MD。
2. **写作风格**：本 Skill 内置完整风格指南（见下文"写作风格"部分），无需加载其他 skill。

### 明确研究对象

拿到用户输入后，确认以下信息。如果用户给得足够明确，不需要追问，直接开始：

1. **研究对象**：具体的产品名/公司名/概念名/人名
2. **类型**：产品、公司、概念、人物、还是其他？
3. **研究动机**（可选）：为什么要研究它？最近发生了什么？
4. **特别关注点**（可选）：有没有特别想深入的方向？

### 分级多轮搜索机制

搜索层采用四轮递进体系，不是一次性撒网。

#### 第一轮：宽泛扫描

**目标：** 快速建立全景认知，发现关键线索、核心人物、重要事件节点。

同时启动 3 个子 Agent，各负责一个方向：

| 子Agent | 职责 | 搜索范围 |
|---------|------|---------|
| **Agent A — 起源与历程** | 研究对象的起源、创始人背景、发展历程概览、关键事件 | 官方文档、Wikipedia、可信媒体报道 |
| **Agent B — 行业与竞争** | 行业背景、市场规模、主要玩家、竞争格局概览 | 行业报告、权威媒体、官方公告 |
| **Agent C — 原理与定位** | 核心技术/方法论/产品原理、定位、目标用户 | 学术论文（如适用）、官方技术博客、产品文档 |

**每个子Agent必须遵循搜索层级约束：**
```
搜索优先级：
1. 官方来源（官网、官方文档、GitHub、官方博客）
2. 权威媒体原创报道（路透、FT、NYT、Bloomberg、36氪原创等）
3. 结构化知识库（Wikipedia、Crunchbase、天眼查基础信息）
4. 学术来源（arXiv、Google Scholar — 仅限技术/学术类对象）
5. 深度分析/行业报告（如需付费全文，找公开摘要和数据点）
6. 社区讨论（Reddit、Hacker News、知乎 — 仅用于用户口碑）

禁止：
- 内容农场、SEO 垃圾站、纯转载站
- 来源不明的自媒体文章
- 未标注日期的信息（无法判断时效性）
```

#### 第二轮：定向深挖

**目标：** 对第一轮发现的关键节点和信息缺口，进行定向深度挖掘。

根据第一轮的输出，自动识别**信息缺口**：

```
信息缺口检测清单：
□ 时间线上有没有明显的中断？（某段时间发生了什么？）
□ 关键决策有没有还原"为什么"？
□ 竞品列表完整吗？有没有遗漏主要玩家？
□ 关键数据有来源支撑吗？（融资额、用户数、市场份额）
□ 用户口碑只看到官方说法还是有真实用户声音？
```

对每个缺口启动专门搜索，使用来源类型特定策略：

| 信息类型 | 优先来源 | 获取方式 |
|---------|---------|---------|
| 学术/技术原理 | arXiv、Google Scholar、ACL/NeurIPS | arxiv API 或 WebFetch 论文页面 |
| 商业数据 | Crunchbase、SEC  filings、企查查、天眼查 | 定向搜索，至少 2 源确认 |
| 用户口碑 | Reddit、Hacker News、知乎、Twitter/X、GitHub Issues | 多平台对比，不依赖单一平台 |
| 融资/收购 | 官方公告 + 财经媒体 | 交叉印证 |
| 技术架构 | 官方技术博客、GitHub Repo、Engineering Blog | 只认官方一手 |
| 市场份额 | Gartner/IDC、上市公司财报 | 标注数据来源和统计口径 |

#### 第三轮：可信度交叉验证 — CRAAP 五维评估

**目标：** 每个关键事实必须经过五维可信度评估，而不是简单贴标签。

应用 **CRAAP 五维评分体系**（时效性、相关性、权威性、准确性、目的性），从五个维度打分：

```
CRAAP 五维评估：

1. Currency（时效性）
   - 信息什么时候发布的？有没有更新？
   - 3年前的数据对历史分析够用，对当前状态则不够
   - 评分：高 / 中 / 低

2. Relevance（相关性）
   - 跟研究对象直接相关还是泛泛的行业背景？
   - 信息深度够支撑分析结论吗？
   - 评分：高 / 中 / 低

3. Authority（权威性）
   - 发布者是谁？有什么资质？
   - 官方公告 > 权威媒体 > 个人博客 > 匿名论坛
   - 评分：高 / 中 / 低

4. Accuracy（准确性）
   - 有具体证据支撑吗（数据、引用、可验证事实）？
   - 与其他来源交叉印证的结果一致还是冲突？
   - 评分：高 / 中 / 低

5. Purpose（目的性）
   - 发布动机是什么？告知、说服、销售、还是娱乐？
   - 有明显的偏见或立场倾向吗？
   - 评分：高 / 中 / 低
```

**五维评分汇总为最终等级（分析过程走五维，报告中呈现等级）：**

```
  🟢 A级 — 五维全部"高" — 官方公告、一手数据、学术论文、经审计财报
  🔵 B级 — 四维"高"或全部"中"以上 — 权威媒体原创报道
  🟡 C级 — 三维"高"或有明显目的性偏差 — 行业分析、资深博主
  🔴 D级 — 时效性或权威性"低" — 论坛讨论、自媒体、传闻
  ⚫ 无法验证 — 标注"未经证实"，不可作为核心论据
```

**交叉验证规则：**

```
A级事实 → 可直接引用，标注来源
B级事实 → 引用时标注来源
C级事实 → 必须找到至少 2 个独立来源印证，否则标注"可信度较低"
D级信息 → 仅用于"用户口碑"部分，不得用于核心分析
无法验证的信息 → 标注"信息暂缺"
所有推测 → 明确标注"这是推测"，必须有逻辑推理过程
每轮评估附一段简短说明，解释为什么给出这个评分
```

#### 第四轮：补漏追根

**目标：** 填补逻辑断层，确保分析不会因为信息缺失而流于表面。

```
补漏信号检测：
□ 纵轴：故事线是否完整？有没有"断了"的感觉？
□ 横轴：主要竞品都分析了？有没有遗漏重要类别？
□ 关键判断有没有信息支撑？还是凭感觉下的结论？
□ 最后检查一遍：有没有某个看似合理的结论其实缺少证据链？
```

有缺则补，补完进入分析层。

## 第二层：分析层 — 四维解构

棱镜分析的分析层按**认知深度递进**展开，而非按分析维度分章。四个维度形成一个完整的因果链：**从起点到分化，从格局到推演。**

### 第一维：起源 — 发展轨迹

这是报告的叙事主体。**核心要求：写一整篇连贯的叙事，不使用任何层级标题（###/####/①/②/③等）。**

每一个关键事件、每一次转折，都必须在叙事中自然铺开，让读者感受到「顺着时间之河流淌而下」的阅读体验。**这一维只聚焦研究对象自己的故事，竞品仅在必要时作为背景出现。**

写作时，你的脑海中要走过以下分析框架，但 **不要把它们暴露为标题或分段**：

---

#### 叙事写作流程

**第一步：确定阶段边界（只在你的脑子里）**

把整个历程自然划分为 3-5 个阶段。阶段之间的分界是某个**重要事件或转折点**。不要在文稿中写"第一阶段"/"第二阶段"，而是在叙事中用过渡句来暗示阶段的转换：

```
阶段转换过渡句示例：
「但2023年刚入春，情况就变了...」
「转折点出现在那场内部会议上。」
「如果说前三年是在搭积木，那2024年就是在盖大楼。」
「XX时代的序幕，是从一次失败的收购拉开的。」
「然而狂奔的背后，裂缝也在悄然出现。」
```

**第二步：按叙事顺序写作，每个事件自然嵌入分析**

不要按「先写事实，再写决策逻辑，再写连锁反应」的固定格式。而是让这些分析维度在叙事中**自然流淌**：

```
叙事中的自然呈现：
                        表层                      深层
「2023年3月，                    —— 事实还原（表层发生了什么）
  他们做了一个看似冒险的决定...  —— 决策逻辑（为什么这么选）

  当时的行业环境是...             —— PEST 自然融入（为什么这个节点是这个选择）

  这个决定放弃了另一条路...      —— 反事实推演（如果选了B会怎样）

  今天回头看，这个决策锁定了...  —— 连锁反应（历史回望）

  表面看是产品策略的胜利，        —— 冰山模型第一层：事件
  更深层的原因是...               —— 冰山模型第三层：结构/驱动力
```

**第三步：每个阶段结束时，自然过渡到下一阶段**

不要用分割线或子标题。用一两句话完成过渡：

```
「这个阶段积累的技术储备和用户口碑，为下一阶段的爆发埋下了伏笔。
只是谁都没想到，引爆点来得这么快——2024年初...」
```

---

#### 要在叙事中隐藏的分析工具

以下工具**不能作为标题或独立段落出现**，必须融入字里行间：

| 工具 | 在叙事中的呈现方式 |
|------|-------------------|
| **PEST 宏观扫描** | 「那一年，政策上...经济上...社会上...技术上...」，自然一段话带过 |
| **三层追问** | 事实陈述 → 「为什么？」→ 「这个选择的影响持续至今」→ 连贯递进 |
| **反事实推演** | 「如果当时选了另一条路...」一句话点明，不展开成独立分析块 |
| **冰山模型** | 「表面看是...但真正驱动这一切的是...」递进式揭示底层逻辑 |
| **阶段划分** | 「但到了X年，风向变了...」→「进入X时代」→ 叙事性过渡 |
| **路径依赖分析** | 「这个决策之所以关键，是因为它让后面的路只剩下了一个方向...」|

---

#### 叙事节奏控制

```
- 每个阶段 3-5 个自然段
- 每段 3-6 句话
- 段落之间跳跃自然：长段→短句单行段→中等段落
- 关键判断单句成段，制造停顿和重量感
- 时间线不要平铺：「2023年做了A，2024年做了B」→ 找到因果关系再写
- 保持"为什么"的追问感，让读者一直带着好奇心往下读
```

---

#### 写作示例

**不要这样写（分层/报告体）：**

```
### 第一阶段：萌芽期（2019-2021）

#### 时代环境
当时 AI 行业正处于...

#### 核心决策
创始团队选择了A方案...
```

**要这样写（叙事体）：**

```
2019年的AI行业，正处于大爆发前夜的宁静。政策上开始鼓励...经济上资本开始涌入...技术上 Transformer 架构刚刚展现出惊人的潜力。就在这样的背景下，四人团队在一个宿舍里敲下了第一行代码。

他们当时面临一个选择：是做一个通用的框架，还是瞄准一个垂直场景。选了前者可能会拿到更大的想象空间，但以他们的资源，在巨头的夹缝中做通用平台无异于以卵击石。他们选了后者。这个选择在后来被证明是明智的——但也在三年后，成了他们最难翻越的那堵墙。

项目很快在开发者社区里传开了。不是因为营销做得好，而是因为用户们发现...（继续叙事）
```

**关键检验标准：** 把写好的时间线部分打印出来（或纯文本浏览），如果还能看出"这是一份报告，有章节结构"，说明分层太明显了，需要重写。如果读起来像一篇深度报道或非虚构特稿，说明对了。

### 第二维：分野 — 岔路口的选择

这是棱镜分析区别于传统分析框架的核心维度。**在关键时间节点上，把研究对象和竞品放到同一帧画面里，看它们分别做了什么选择、为什么走向了不同方向。**

它不是"先讲完历史再讲竞品"的并列关系，而是把时间线和竞品对比**融合**在一个维度中——过去同一个时间点，A选了X、B选了Y，由此产生了今天的能力差异。

```
分野分析的三个步骤：

第一步：识别关键岔路口
  从研究对象的历程中，选出 3-5 个关键节点
  标准：这些节点上存在明显的选择分歧——研究对象和竞品走了不同的路
  特别关注：技术路线选择、市场定位博弈、商业模式分叉

第二步：还原每个岔路口的多方视角
  在时间T——
  研究对象做了什么选择？为什么？
  同期竞品A做了什么选择？为什么？
  竞品B呢？
  当时的行业环境是？（PEST 自然融入）

第三步：判断路径差异的后果
  这些不同的选择，如何导致了今天的能力差异？
  如果当初选了另一条路，今天的格局会有什么不同？
  研究对象的某个优势，能追溯到哪个岔路口的决策？
  研究对象的某个劣势，又是哪次选择的代价？
```

#### 分野章节的写作方式

```
每个岔路口是一个独立的叙事单元（不出现"岔路口1/2/3"标题）：

  2023年初，整个行业都在关注AI编程这个方向。

  研究对象的反应是...它选择了A路径。
  当时它面临的约束是...它的判断逻辑是...

  同一时间，竞品A的判断完全不同——它选了B路径。
  它之所以这么选，是因为它的基因/资源/认知决定了...

  这个差异在当时看起来不致命，但现在回头看——
  一个走向了...另一个走向了...
  如果当时选了另一条路...（反事实推演）

（自然过渡到下一个岔路口）
```

**关键区别：** 第一维"起源"讲的是研究对象自己的时间线；第二维"分野"把竞品拉到同一时间点上做并排对比。两者在写作上是承接关系——先让读者了解研究对象发生了什么，再让读者理解"为什么它和竞品不一样"。

---

### 第三维：格局 — 当前的位势

经过"起源"和"分野"的分析，读者已经理解了"为什么它们长成了今天这样"。现在给出一个**纯横截面的全景快照**——此时此刻，各方在产品、技术、市场、商业模式上的位置对比。

#### 确定分析范围

根据竞争格局的密度，分三种场景处理：

- **场景A（无直接竞品）** → 分析壁垒、替代方案、潜在竞争者
- **场景B（少量竞品 1-2 个）** → 逐一深入对比
- **场景C（竞品充分 3+）** → 选 3-5 个代表性竞品，其余简要提及

#### 商业模式画布 — 竞品画像的标准工具

在开始对比之前，先用**商业模式画布**的九宫格对每个主要竞品做结构化拆解。**输出时仍是叙事文本**，画布是作者的分析工具，不给读者看格子。

```
商业模式画布九宫格（作者在脑中填写）：

┌──────────────────────────────┬──────────────┬──────────────────────────┐
│         KP 合作伙伴          │   KA 关键业务  │       VP 价值主张         │
│  谁在帮它？依赖谁？           │  它在做什么？  │  用户为什么选它？          │
├──────────────────────────────┼──────────────┼──────────────────────────┤
│                              │   KR 核心资源  │                          │
│                              │  它有什么别人没有的？                     │
├──────────────────────────────┼──────────────┼──────────────────────────┤
│         CR 客户关系           │    CH 渠道     │        CS 客户细分        │
│  怎么留住用户？               │  怎么触达用户？ │  目标用户是谁？            │
├──────────────────────────────┴──────────────┴──────────────────────────┤
│                              C$ 成本结构                               │
│                        VS                                              │
│                              R$ 收入来源                               │
│  它在赚钱还是烧钱？怎么赚钱/花钱？                                     │
└───────────────────────────────────────────────────────────────────────┘
```

**把画布转化为叙事的方式：**
```
不要写「目标用户是...，价值主张是...，收入来源是...」
要写「它的用户画像很清晰——...。吸引这些用户的核心理由是...。
它通过...来触达用户，靠...留住用户。成本大头在...，但目前收入主要来自...，
算下来...所以它（是/不是）一个能自我造血的模式。」
```

**每分析完一个竞品，自然引出与研究对象的差异：**
```
「相比之下，研究对象的用户群体完全不一样...」
「两者的本质区别在于，一个在...，另一个在...」
```

#### 三维竞争分析框架

#### 重点优势与劣势溯源

基于前三个维度的分析，汇总两项核心判断：

```
今天的优势，根在过去的哪个岔路口？
  ├── 优势1 → 追溯至 [具体时间/决策]
  ├── 优势2 → 追溯至 [具体时间/决策]
  └── 是否可持续？

今天的劣势，根在过去的哪个岔路口？
  ├── 劣势1 → 追溯至 [具体时间/决策]
  ├── 劣势2 → 追溯至 [具体时间/决策]
  └── 是否可逆转？
```

---

### 第四维：推演 — 变量与场景

基于前三维（起源→分野→格局）的完整分析，面向未来识别关键不确定性，构建场景推演。

不写"最可能/最危险/最乐观"三个固定剧本，而是：

```
第一步：识别关键变量
├── 哪些变量当前最不确定？
├── 技术突破？政策变化？竞争对手动作？用户偏好迁移？
└── 每个变量的变化方向和可能性

第二步：变量组合构建场景
├── 选 2-3 个关键变量
├── 不同走向排列组合出 3-4 个场景
└── 每个场景要有逻辑推演链，不是拍脑袋

第三步：每个场景的完整分析
├── 场景描述：这个场景下会发生什么？
├── 逻辑链条：为什么这个场景是可能的？
├── 早期信号：如果这个场景正在发生，会有什么迹象？
└── 行动建议：研究对象可以提前做什么？

第四步：概率与判断
├── 基于当前信息，每个场景的大致概率
├── 不追求精确数字，但要给出相对可能性判断
└── 标注"我的判断是"，承认主观性
```

---

## 第三层：输出层 — 叙事呈现

### 写作风格总则

棱镜分析的写作风格是它的灵魂。目标是：**用报告的信息密度，用叙事的阅读体验。**

这不是一份冷冰冰的咨询报告，而是一篇让人能从头读到尾的深度研究。

#### 核心原则

**叙事驱动，不是罗列驱动。**
- 时间线部分要有故事弧线，有起承转合
- 不要写成"2023年1月A发布，2023年3月B发布"的流水账
- 每个事件的因果关系要讲清楚——"A发生了，因为B；然后导致了C"

**知识是聊着聊着顺手掏出来的。**
- 在讲述过程中自然地带出背景知识
- 不要"下面我来给大家科普一下XX"

**敢下判断，但必须有事实支撑。**
- 先摆事实，再给判断
- 是推测的明确标注
- 判断用"我觉得""我的判断是"的姿态，不是居高临下的定论

**层层剥开，而不是直奔结论。**
- 用"现象→表面解释→更深的追问→核心洞察"的方式展开
- 让读者参与到思考过程中

**回环呼应。**
- 开头或时间线部分埋的细节和钩子，在交汇洞察或结尾callback回来
- 前后因果的闭合感，让报告从"信息流"变成"作品"

#### 金字塔结构（在写作前先搭框架）

在动笔写报告之前，先用**金字塔原理**搭好整篇报告的逻辑骨架。这不是写给别人看的，是给你自己用的。

```
金字塔搭建步骤：

第一步：写出核心结论
  └─ 用一句话说清楚：这个研究的最终判断是什么？
  └─ 这句结论就是金字塔的塔尖，也是报告给读者的第一印象

第二步：列出支撑结论的 3-5 个关键论据
  └─ 它们是金字塔的第二层
  └─ 起源分析揭示了什么规律？
  └─ 分野对比暴露了什么差异？
  └─ 格局与推演给出了什么判断？

第三步：每个论据向下拆解事实
  └─ 第三层：支撑每个论据的具体事实、数据、案例
  └─ 这些事实应该归类分组（MECE：相互独立、完全穷尽）
  └─ 同一组的事实按逻辑递进排列

第四步：检查逻辑
  └─ 结论 ← 论据 ← 事实：每一层都能经得起"所以呢？"的追问
  └─ 事实 ← 论据 ← 结论：每一层都能经得起"为什么？"的追问
```

金字塔搭完之后，**按叙事的方式写出来**——让读者从第一句话读到最后一句话，感受到逻辑的层层递进，而不需要看到金字塔本身。

#### 语言层面的具体要求

**好句子：**
- 句子时长时短，段落之间跳跃自然
- 一句话自成一段制造重量感
- 每段都有"核"，其他句子围绕它展开

**不要出现的东西：**
- "首先...其次...最后"
- "值得注意的是" / "不难发现"
- "赋能" / "抓手" / "打造闭环" / "对齐"
- "在当今XX快速发展的时代" / "随着技术的不断进步"
- "说白了" / "意味着什么？" / "这意味着" / "本质上" / "换句话说" / "不可否认"
- 不说"AI工具"、"某个模型"，要说具体名字
- 编造场景——搜不到就标注"该信息暂缺"

**具体，不要抽象。**
- 不要写"该公司实现了快速增长"
- 要写"从2024年中期的1000万美元ARR到2025年底的10亿美元，增长曲线几乎是垂直的"

封面页（PDF自动生成）

目录

一、一句话定义

#### PDF 生成

使用本 Skill 自带的 `scripts/md_to_pdf.py` 脚本将 Markdown 转为排版精美的 PDF。

**转换流程：**
```bash
pip install weasyprint markdown --break-system-packages
python [skill目录]/scripts/md_to_pdf.py input.md output.pdf --title "研究对象名称" --author "棱镜分析"
```

如果 WeasyPrint 不可用（如 Windows 环境），则只输出 Markdown 文件，在报告开头说明。

#### 文件命名

- HTML：`[研究对象]_棱镜分析报告.html`
- Markdown：`[研究对象]_棱镜分析报告.md`
- PDF：`[研究对象]_棱镜分析报告.pdf`

#### HTML 生成（必选，每份报告必须生成）

HTML 是棱镜分析的标准输出格式。每份报告生成时，必须同时输出一份独立的 HTML 文件。

**HTML 样式要求：**
- 使用衬线字体（Noto Serif SC 或系统衬线），阅读体验接近印刷品
- 米白底色（#f5f3ee / #f7f5f0），文字深灰（#1a1a1a），行距 1.8-2.0
- 一句话定义使用左边框强调样式（非黑底框），与整体调性统一
- 关键判断使用左边框浅色背景块区分
- 可信度标签使用小圆角彩色标签，上标对齐
- 表格使用深色表头 + 斑马纹行
- 场景卡片使用彩色概率标签（绿/黄/红）
- 内联 CSS，不依赖外部 CDN（确保离线可查看）
- 响应式设计，移动端适配

**输出流程：**
1. 先写 Markdown → HTML（手写或脚本转换，保内联样式）
2. 如果 WeasyPrint 可用，再转 PDF
3. HTML 和 MD 为必选产出，PDF 为可选

- [ ] 没有触犯语言禁区里的任何一条？
- [ ] 写作风格有节奏感、有可读性？
- [ ] **时间线部分没有出现层级标题**（###/####/①/②），是一整篇叙事？
